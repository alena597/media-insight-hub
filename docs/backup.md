# Резервне копіювання та відновлення (Release engineer / DevOps)

Проєкт зберігає стан користувачів, історії та обраного у файлі `server/data.json` (шлях задається змінною `DATABASE_PATH`). Класичної СУБД немає — резервування зводиться до копіювання файлів та конфігурації.

## 1. Стратегія резервного копіювання

- Повні копії каталогу даних і конфігурації перед кожним оновленням і щодня (за навантаженням).
- Зберігати копії поза сервером додатку (інший диск, S3-сумісне сховище, окремий хост).
- Перед руйнівними операціями — обов’язкова ручна копія `data.json`.

## 2. Типи резервних копій

| Тип | Опис | Застосування |
|-----|------|--------------|
| Повна | Копія всього релевантного набору (`data.json`, `.env`, опційно весь `server/`) | Щоденно / перед оновленням |
| Інкрементальна | Зміни від останньої повної (rsync, backup-інструменти) | Великі обсяги, зменшення часу |
| Диференціальна | Зміни від фіксованої повної копії | Рідше в малих проєктах |

Для одного JSON-файлу часто достатньо повної копії з ротацією за датою.

## 3. Частота та ротація

- Критичні дані (`data.json`): щоденно мінімум; за високою активністю — кілька разів на день.
- Ротація: зберігати останні 7–30 днів щоденних копій + місячні «довгі» копії.
- `.env`: при кожній зміні секретів — нова зашифрована копія в сейфі (не в git).

## 4. Що саме копіювати

| Об’єкт | Шлях | Примітка |
|--------|------|----------|
| Користувацькі дані | `server/data.json` або `DATABASE_PATH` | Основний актив |
| Конфігурація | `server/.env` | Секрети; не комітити |
| Логи | За налаштуванням journald/PM2 | Опційно, для розслідувань |
| Код | Git remote | «Джерело правди» — репозиторій, не знімок диска |

## 5. Процедура резервного копіювання (ручна)

На Linux:

```bash
BACKUP_ROOT=/backup/mih
DATE=$(date +%Y%m%d_%H%M)
mkdir -p "$BACKUP_ROOT/$DATE"
cp /opt/mih/server/data.json "$BACKUP_ROOT/$DATE/data.json"
cp /opt/mih/server/.env "$BACKUP_ROOT/$DATE/env.backup"
chmod 600 "$BACKUP_ROOT/$DATE/env.backup"
```

Перевірка:

```bash
ls -la "$BACKUP_ROOT/$DATE"
```

## 6. Перевірка цілісності

- Порівняти розмір файлу з попередньою копією.
- Відкрити JSON у валідаторі або `node -e "JSON.parse(require('fs').readFileSync('data.json','utf8'))"`.
- Періодично відновлювати копію на тестовому середовищі і піднімати API.

## 7. Автоматизація

- cron + скрипт копіювання (див. [`scripts/backup-example.sh`](scripts/backup-example.sh)).
- Хмарні бекапи диска VM (snapshots) — додатковий рівень, не заміна логічного бекапу `data.json`.

## 8. Відновлення з резервної копії

### 8.1. Повне відновлення стану даних

1. Зупинити API: `systemctl stop mih-api`.
2. Скопіювати резервний `data.json` на місце робочого файлу.
3. Вирівняти права доступу (`chown` користувача сервісу).
4. Запустити API: `systemctl start mih-api`.
5. Перевірити `/api/health` і вхід тестового користувача.

### 8.2. Вибіркове відновлення

- Якщо потрібно повернути лише одного користувача — вручну відредагувати JSON (ризиковано) або відновити файл на staging, експортувати потрібні записи, злити вручну. Для продакшену краще мати повну копію на момент до пошкодження.

### 8.3. Тестування відновлення

- Раз на квартал виконувати навчальне відновлення на ізольованій VM.
