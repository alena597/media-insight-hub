# Резервне копіювання та відновлення (Release engineer / DevOps)

Проєкт зберігає стан користувачів, історії та обраного у SQLite-файлі `server/data.db` (шлях задається змінною `DB_PATH`). Резервування зводиться до копіювання файлу бази даних та конфігурації.

> **WAL-режим:** при активній роботі API поряд із `data.db` присутні файли `data.db-shm` та `data.db-wal`. Для коректного бекапу потрібно або зупинити API, або використати вбудований механізм SQLite (`.backup`), або скопіювати всі три файли разом.

## 1. Стратегія резервного копіювання

- Повні копії бази та конфігурації перед кожним оновленням і щодня (за навантаженням).
- Зберігати копії поза сервером додатку (інший диск, S3-сумісне сховище, окремий хост).
- Перед руйнівними операціями — обов'язкова ручна копія `data.db`.

## 2. Типи резервних копій

| Тип | Опис | Застосування |
|-----|------|--------------|
| Повна | Копія `data.db` (зупинений API або `.backup`) + `server/.env` | Щоденно / перед оновленням |
| SQL-дамп | `sqlite3 data.db .dump > backup.sql` — текстовий SQL-скрипт | Міграція між серверами, довготривале зберігання |
| Знімок диска | Snapshot VM/тому | Додатковий рівень, не заміна логічного бекапу |

## 3. Частота та ротація

- `data.db`: щоденно мінімум; за високою активністю — кілька разів на день.
- Ротація: зберігати останні 7–30 днів щоденних копій + місячні «довгі» копії.
- `.env`: при кожній зміні секретів — нова зашифрована копія в сейфі (не в git).

## 4. Що саме копіювати

| Об'єкт | Шлях | Примітка |
|--------|------|----------|
| База даних | `server/data.db` (або `DB_PATH`) | Основний актив |
| WAL-файли | `server/data.db-shm`, `server/data.db-wal` | Копіювати разом із `data.db` при гарячому бекапі |
| Конфігурація | `server/.env` | Секрети; не комітити |
| Логи | `server/logs/` | Опційно, для розслідувань |
| Код | Git remote | «Джерело правди» — репозиторій, не знімок диска |

## 5. Процедура резервного копіювання

### 5.1. Холодний бекап (API зупинено — найнадійніший варіант)

```bash
sudo systemctl stop mih-api

BACKUP_ROOT=/backup/mih
DATE=$(date +%Y%m%d_%H%M)
mkdir -p "$BACKUP_ROOT/$DATE"
cp /opt/mih/server/data.db "$BACKUP_ROOT/$DATE/data.db"
cp /opt/mih/server/.env "$BACKUP_ROOT/$DATE/env.backup"
chmod 600 "$BACKUP_ROOT/$DATE/env.backup"

sudo systemctl start mih-api
```

### 5.2. Гарячий бекап (API працює) — через SQLite online backup

```bash
sqlite3 /opt/mih/server/data.db ".backup /backup/mih/$(date +%Y%m%d_%H%M)/data.db"
```

Команда `.backup` використовує внутрішній API SQLite і є безпечною під час активних транзакцій.

### 5.3. SQL-дамп (для міграцій та архіву)

```bash
sqlite3 /opt/mih/server/data.db .dump > /backup/mih/$(date +%Y%m%d_%H%M)_dump.sql
```

Перевірка:

```bash
ls -la "$BACKUP_ROOT/$DATE"
```

## 6. Перевірка цілісності

- Порівняти розмір файлу з попередньою копією.
- Перевірити цілісність: `sqlite3 /backup/mih/<DATE>/data.db "PRAGMA integrity_check;"` — має повернути `ok`.
- Або перевірити через Node.js: `node -e "import('better-sqlite3').then(({default:D})=>{ const db=new D('/backup/.../data.db'); console.log(db.prepare('SELECT COUNT(*) as c FROM users').get()); db.close(); })"`.
- Періодично відновлювати копію на тестовому середовищі і піднімати API.

## 7. Автоматизація

- cron + скрипт копіювання (див. [`scripts/backup-example.sh`](scripts/backup-example.sh)).
- Хмарні бекапи диска VM (snapshots) — додатковий рівень, не заміна логічного бекапу.

## 8. Відновлення з резервної копії

### 8.1. Повне відновлення стану даних

1. Зупинити API: `systemctl stop mih-api`.
2. Скопіювати резервний `data.db` на місце робочого файлу.
3. Видалити залишкові WAL-файли (якщо є): `rm -f data.db-shm data.db-wal`.
4. Вирівняти права доступу: `chown www-data:www-data data.db`.
5. Запустити API: `systemctl start mih-api`.
6. Перевірити `/api/health` і вхід тестового користувача.

### 8.2. Відновлення з SQL-дампу

```bash
sqlite3 /opt/mih/server/data.db < /backup/mih/<DATE>_dump.sql
```

### 8.3. Вибіркове відновлення

Якщо потрібно повернути лише окремі записи — відновіть базу на staging, виконайте потрібні SQL-запити, перенесіть результат вручну. Для продакшену краще мати повну копію на момент до пошкодження.

### 8.4. Тестування відновлення

- Раз на квартал виконувати навчальне відновлення на ізольованій VM.
